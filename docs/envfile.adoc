= Envfile Configuration
:toc:
:toclevels: 2

The `envfile` property provides advanced control over environment variable loading and manipulation.

== Basic Usage

[source,yaml]
----
tasks:
  my-task:
    envfile:
      path: ./my.env
    command: echo $MY_VAR
----

== Envfile Properties

[cols="2,1,3",options="header"]
|===
| Property | Type | Description

| `path`
| string or array
| Path(s) to environment file(s)

| `exclude`
| array
| Variable name prefixes to exclude

| `include`
| array
| Variable names to include (whitelist)

| `modify`
| array
| Key transformation rules (see <<Key Modification>>)

| `replace_char`
| string
| Separator character used when writing envfiles (default: space)

| `quote`
| boolean
| Whether to quote values in generated envfiles

| `generated_dir`
| string
| Directory for generated envfiles (default: `.eirctl`)

|===

== Path Configuration

=== Single File

[source,yaml]
----
envfile:
  path: ./.env
----

=== Multiple Files

Later files override earlier ones:

[source,yaml]
----
envfile:
  path:
    - ./base.env       # Loaded first
    - ./override.env   # Overrides base.env
----

=== Precedence

Multiple `envfile.path[]` can be specified at different levels with this order of precedence:

----
context < task < stage (task called from a pipeline)
----

== Include and Exclude

=== Exclude Variables

Prevent specific variables from being injected:

[source,yaml]
----
contexts:
  container:
    container:
      name: alpine:latest
    envfile:
      exclude:
        - PATH
        - HOME
        - TMPDIR
        - GOPATH
        - SHELL
----

=== Include Variables (Whitelist)

Only include specified variables:

[source,yaml]
----
envfile:
  path: ./all-vars.env
  include:
    - DATABASE_URL
    - API_KEY
    - LOG_LEVEL
----

== Key Modification

Transform variable names using `modify`:

[source,yaml]
----
envfile:
  path: ./.env
  modify:
    - pattern: "(?P<keyword>)(?P<varname>.*)"  # Regex with named groups
      operation: upper                          # Convert to uppercase
----

=== Operations

[cols="1,3,2",options="header"]
|===
| Operation | Description | Example

| `upper`
| Convert matching variable names to uppercase
| `my_var` → `MY_VAR`

| `lower`
| Convert matching variable names to lowercase
| `MY_VAR` → `my_var`

|===

IMPORTANT: The regex pattern *must* contain two named capture groups: `keyword` and `varname`.

=== Examples

[source,yaml]
----
envfile:
  path: ./.env
  modify:
    # Convert all variable names to uppercase
    - pattern: "(?P<keyword>)(?P<varname>.*)"
      operation: upper

    # Convert all variable names to lowercase
    - pattern: "(?P<keyword>)(?P<varname>.*)"
      operation: lower
----

== In-File Variable References

Environment files support referencing previously defined variables:

[source,env]
----
# base-url.env
BASE_URL=https://api.example.com
API_ENDPOINT=${BASE_URL}/v1
FULL_URL=$BASE_URL/v1/users
----

NOTE: Referenced variables must be defined earlier in the file (files are scanned line by line).

== Generated Envfile

For executable contexts using Docker CLI, a generated envfile can be placed in a specific directory:

[source,yaml]
----
contexts:
  docker-exec:
    executable:
      bin: docker
      args:
        - run
        - --rm
        - --env-file
        - envfile
    envfile:
      generated_dir: /tmp/eirctl
      exclude:
        - HOME
        - PATH
----

NOTE: Native container contexts handle environment injection automatically and don't need this.

== Context-Level Configuration

Set default envfile configuration for all tasks using a context:

[source,yaml]
----
contexts:
  production:
    container:
      name: myapp:latest
    envfile:
      path: ./production.env
      exclude:
        - PATH
        - HOME
        - TMPDIR
----

Tasks using this context inherit the envfile configuration.

== Auto-Loading eirctl.env

If an `eirctl.env` file exists in the working directory, it is automatically loaded at the context level.

[source,env]
----
# eirctl.env (auto-loaded)
DATABASE_URL=postgres://localhost/mydb
REDIS_URL=redis://localhost:6379
----

This follows the standard precedence: `Context < Pipeline < Task`

== Practical Examples

=== Multi-Environment Setup

[source,yaml]
----
tasks:
  deploy-staging:
    envfile:
      path:
        - ./base.env
        - ./staging.env
      exclude:
        - LOCAL_ONLY_VAR
    command: ./deploy.sh

  deploy-production:
    envfile:
      path:
        - ./base.env
        - ./production.env
      exclude:
        - LOCAL_ONLY_VAR
        - DEBUG_MODE
    command: ./deploy.sh
----

=== Variable Name Transformation

[source,yaml]
----
# Convert variable names to uppercase
envfile:
  path: ./aws.env
  modify:
    - pattern: "(?P<keyword>)(?P<varname>.*)"
      operation: upper
----

=== Clean Container Environment

[source,yaml]
----
contexts:
  clean-alpine:
    container:
      name: alpine:latest
    envfile:
      exclude:
        # System paths
        - PATH
        - HOME
        - TMPDIR
        - SHELL
        # Language-specific
        - GOPATH
        - GOROOT
        - NODE_PATH
        # Editor/IDE
        - EDITOR
        - VISUAL
        - TERM
----

=== Secrets Management

[source,yaml]
----
tasks:
  deploy:
    envfile:
      path: ./secrets.env
      include:          # Whitelist only needed secrets
        - DATABASE_URL
        - API_KEY
        - JWT_SECRET
    command: ./deploy.sh
----
